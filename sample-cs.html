The algorithm that taught me to slow down

Dynamic programming has a reputation for being hard. I think this is because people approach it the wrong way — they try to figure out the transitions before they understand the subproblems. They want to see the whole staircase before they've found the first step.

The idea is simple: a problem has optimal substructure if its optimal solution can be built from optimal solutions to smaller instances. And it has overlapping subproblems if those smaller instances repeat. When both conditions hold, you memoize — you remember — and an exponential algorithm becomes polynomial.

The canonical example

Fibonacci. Naively:

def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

This runs in $O(2^n)$. Every call fans into two more, most of which repeat work already done. With memoization:

def fib(n, memo={}):
    if n in memo: return memo[n]
    if n <= 1: return n
    memo[n] = fib(n-1, memo) + fib(n-2, memo)
    return memo[n]

Now it's $O(n)$. Each subproblem is solved exactly once. The rest is just lookup.

What this teaches

The lesson isn't about tables. It's about trust in decomposition. Before you can solve $n$, you need $n-1$ and $n-2$. Before you write the chapter, you need the outline. Before you read the paper, you need the prerequisites.

Restlessness wants to skip to $n$. It wants the result without the path. Dynamic programming is a formal proof that you cannot. The recurrence demands the base cases. The tower demands the foundation.

Slow down. Solve the small things. Trust the recurrence.
